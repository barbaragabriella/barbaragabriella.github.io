What is this project about?

Smart and Safe Home (SSH) was created with the porpuse to control lighting, air conditioning inside home and secure the front door/gate against intruders as well as monitoring home in case of fire. The aim is to build a smart home with focus on safety and energy consumption.
If we control every light and air conditioner in a home, we will make sure no energy will be wasted just because someone forgot to turn off lights/air conditioner when left home. Also, the system will make sure only allowed individuals will enter the house through face recognition. Whenever someone not allowed tries to be recognized, the system will alert the owner sending an intruder alert with a picture.
To make sure the user will be able to interact with the system, we also developed a mobile app which will allow the user to turn on the lights before getting home or turn them off when leaving and set up a temperature for the air conditioner. Also, the user will be able to monitor the presence, temperature, humidity and smoke data.

System Architecture

The architecture of the system here proposed is shown in below. As a project criteria, all data should be stored in a cloud storage. API (Application Programming Interface) is responsible for the data management. It controls the data flow coming from and going to the device and mobile app. In other words, the data goes through the API to be stored in the cloud, to reach the app mobile and also any info that goes to the device.



Heroku

Heroku is a cloud platform as a service that supports several programming languagues (Noje.js, Python, Java ...). It lets you deploy, run and manage applications. An application is a collection of source code that instructs a build system as to which additional dependencies are needed in order to build and run the application. You can know more about how Heroku works and how to use it, here.
Heroku was used in this application to host our API (developed using Node.js). It is responsable for controlling data flow in and out the devices/mobile app. So, whenever a data had to be sent to the database (POST), or get a data from database (GET), or even update an existing data (PUT) it would go through the API to the database regardless of the data source.

You can find the source node.js code used in this project here.

Firebase Realtime Database

For cloud storage, we are using Firebase platform to store and sync data in realtime. Firebase Realtime Database is a cloud-hosted NoSQL database, it is really simple to use and meets all requirements regarding storage for this project. You can get more details about this platform here. Our database is shown below. We have a room (home) with control, humidity, intruder, light, presence, set temperature, smoke and presence nodes. Take a look at what each node means:

Control: To store if the user wants to control or not the lights. The command comes from mobile app
Light: Once the user enables the control mode, he/she will be able to turn on/off lights using the mobile app
Set Temperature: If the air conditioner is on, the user will be able to set up its temperature by choosing from mobile app
Intruder: It will store intruder data. Whenever someone that is not registered tries to be regonized, the device (Raspberry Pi) will send this data to the cloud. Also, it will send an e-mail with a picture of the intruder to the user so he/she will be aware of the situation
Humidity: Stores humidity data that comes from the device (ESP8266)
Temperature: Stores temperature data that comes from the device (ESP8266)
Smoke: Stores smoke data that comes from the device (ESP8266). This data is only sent to the cloud if the sensor detects fire, otherwise it will be empty (let's hope it never gets any data)
Presence: Stores presence data that comes from the device (ESP8266). Whenever a presence is detected inside the home, the device will send to be stored in the cloud

Hardware Components

Sensors

Motion sensor (PIR): PIR is used to detect the motion of human inside a room. The sensor uses infrared rays to map an initial pattern for reflected rays of surrounding objects. Once the calibration is done after activating the sensor, any change in the previously recognized pattern would set the digital output of the sensor accordingly (1: motion detected, 0: no motion).

Temperature and Humidity sensor (DHT22): DHT22 is digital temperature (T) and humidity (RH) sensor. The sensor requires a library defined in MPU to communicate through a serial link to provide continues reading at frequency managed by the MPU.

Gas concentration sensor (MQ-2): MQ-2 is used for smoke detection. The sensor provides an analog output voltage proportional to CO and CO2 concentration in ambient air. If a sensitivity threshold manually programmed by the user is exceeded, the MPU recognizes the smoke presence and sounds the fire alarm.

Actuators

Light Emitting Diodes (LEDs): LEDs are used as indicators for actuation events. In the illustrated system below, a variety of LEDs reflect different functions of the system.
• The yellow LED represents the lighting inside the room. At any given instant, room light is controlled by either the user app or human presence. The priority is always given to user commands which interrupt the operation of the light. Once the user drops direct control, the light follows the most updated status relevant to the motion sensor. If human presence is detected at that time the light would be active. When no human motion is detected there are two possibilities depending on a countdown that is initiated after the lack of human presence. If a delay of 30 seconds has passed the light is turned off. Otherwise, the light is set to be on until the waiting period ends.
• The green and LEDs are temperature indicators, the red LED is active if the temperature is greater than 20oC. Otherwise, the green LED is activated. Both these LEDs operate when motion is detected or until 30 seconds have passed since last motion detection.
• The RGB NeoPixel resembles an air conditioning unit (AC). The AC operation temperature is represented by the color of the NeoPixel, a scale of colors between (Hot: red and Blue: cold). Temperature is set manually by the user via the mobile app at any given time. The NeoPixel is controlled like the yellow LED with an added constraint of the ambient stored temperature being greater than 20oC.
• The red fire alarm LED remains on along with the buzzer if smoke is detected.
Buzzer: The buzzer is used as a fire alarm activated in case of smoke detection. The alarm is turned off either manually or automatically when no more is detected in the premises.

Communication with Database

There are several key features that need to be taken into consideration when establishing connections for communication purposes and accumulating sensory measurements. Data processing and storage space, transmission power time minimization and network congestion mitigation. These features were sought out to be met through the investigation of adequate intervals to achieve enhanced performance for sending sensory data and receiving user commands. The investigation is based on the function and operation of each type of sensors. Sensors responsible for detection events like motion and smoke sensors measure the premises all the time but only record their findings once a status change occurs. Otherwise, the database would be flooded with redundant detection data. For pressure and humidity sensor, a record of ambient conditions is sent once every 15 minutes. A resolution that is found convenient to keep control over the environment. In the same reasoning, user control preferences are updated all the time to achieve real-time operation schemes.
Sending and receiving data is managed using JSON syntax structured commands. A connection is established each time a detection and/ or a change of status is confirmed for a sensor – in the case of the motion and smoke sensors, or it is time already to send data - in case of temperature and humidity sensors. A JSON message is initiated by creating a unique JSON buffer object with a nested value object. Upon writing the message with the specified syntax, the message is sent using http “post” command with a specified header and path.

To read control values in real time from database. A special database library “get” command from Firebase is used with specific path and type for each value needed to be received.

Tests

A variety of tests for system operation conditions – Internet connection status, presence detection, ambient temperature and user control command – have been performed to test the general performance and expected response of the system. Functionality duration

The system acquires user control status and commands by a database provided library (Firebase). Status update is checked for each loop by a get command for various user entries, this is not an optimized way keeping real-time control of the system. After an approximated time found to be 30 minutes the database shuts down the connection due to excessive get commands.

Standalone Operation

The system is fully capable of maintaining safe and smart (predefined) responses in case of losing internet connections. The MCUs continue to operate under preset operation conditions and programs to keep secure home entrance.

Future Development

Development of the current system can be sought out for two major aspects of system operation.

Security

The concept of security can be improved in two aspects. Firstly, the face recognition system can ask the user to provide a confirmation to open the door in case a registered user is granted access at the entrance. This is to avoid a possible security breach in the form of providing a user personal photograph instead of the actual person presence. Secondly, a secure internet connection protocol – like HTTPS – must be inducted; this is to avoid any possible spying or hacking attempts to the system components.

Internet Connection

Status change of user control inputs needs to be established by means of a socket connection. This type of connection provides optimized means of communication between the server and the MCU. The server manages informing any updates on the current control status. In other words, the localized control keeps listening in any given time for a status change without data transition actions. The server only informs the system of a change in the case of user control update then the system acts accordingly. This has been attempted but socket connection was not established properly, instead the alternative approach of continuous data request using the database library was adopted.

Raspberry Pi

We already know how to control lights and air conditioner and how to monitor smoke data. Now we will go through the safety part. We are trying to control the entrance by only open de door/gate when someone from the house is trying to get in. In other words, we are trying to recognize authorized individuals. To do that, we will be doing a face recognition that thanks to OpenCv it is easier than it sounds.
There are basically 4 steps to do a face recognition:
• Gather data: Gather images of the person (face only) that you want to recognize
• Train recognizer: Teach the recognizer with the photos you have chosen so it can learn who to recognize
• Recognition: Show some new faces to the recognizer to see if it is working the way it should
• Use your data: Now you do your logic with the faces recognized.

OpenCV has three built in face recognizers and you only need one code line to use them. In this project, we are using Local Binary Patterns Histograms (LBPH) Face Recognizer which goes but this calling: cv2.face.createLBPHFaceRecognizer().
Eigenfaces and Fisherfaces are the other two face recognizers available on OpenCv but they are both affected by light and in real life we cannot guarantee perfect light conditions. LBPH face recognizer is an improvement to overcome this drawback. The idea behind LBPH is to not look at the image as a whole instead find the local features of an image. LBPH alogrithm try to find the local structure of an image and it does that by comparing each pixel with its neighboring pixels. You can find more about this method here.

Because we will be constantly monitoring the entrance, we will be using a Raspberry Pi Camera along with the Raspberry Pi. This part was developed using Python, best programming language to work with specially when dealing with image processing.
To start, we need some pictures to train the recognizer. Around 12 pictures would be a good start.
Once you have your dataset, you train your recognizer. Here you can find our folder with a dataset which you have to fill with your pics, the detectors, trainer and main code (face_recognition.py). You will find the complete code there and you can try yourself only adding your own info, email and API url.
The trainer will train your recognizer. You only need to execute (python trainer.py) after putting your pics in the dataset. If you want to recognize more than one person, just add more folders following the sequence number.
Once the trainer is done, you will get a file named trainedData.yml. It will allow your main code to access your recognizer so you do not need to train everytime you run your main code.

Going through the main code you will see there are some libraries (OpenCv, time, numpy, picamera and others). Most of these library can be installed used the command pip install library-name. For example, to install numpy, you can run pip install numpy. To install OpenCv, this command should be executed on command line sudo apt-get install python-opencv. Below the libraries on the main code, there is a function responsable for sending the email to the owner. So whenever an intruder is detected, the owner will receive an e-mail with a pic only once, because we want to make sure your mailbox will not receive tons of e-mails. Also, we send the data to our cloud storage.
After the e-mail function, we load our detector and then the real code starts. The main code is well comented so you will find all information you need to understand it. Whenever we detect someone, a square is drawn around the face with either the recognized face's name or an intruder tag. If you are in the dataset, an "Access Granted" will be displayed after a while. Otherwise, it will display an "Intruder" message and send the e-email.



Limitations

One of the big limitations most projects regarding image processing have faced so far is to know exactly if the image is real. For example, in this project we are trying to recognize faces but there could be a real face, a picute in a mobile phone or even a printed one. In all of those cases, if the resolution is good enough, the program would recognize a face and tell if it is someone known or unknown. This issue decreases the safety method once someone can just show a picture and get inside of the house.
We did not go that far to try to solve this issue. However, there might br some option to increase the efficiency such as using machine learning. In this case, it would be possible to first detect if there was a real person in the image and then recognize the face. Another possibility could be detect what is around the face, if there are phone's edge, paper's edge and an infinity of possibilities that might arise.
